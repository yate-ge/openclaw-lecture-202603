# 用 Claude Code 做 Slides：一个Openclaw主题分享课件的制作实践



**在线预览**: [GitHub Pages](https://yate-ge.github.io/openclaw-lecture-202603/) ｜ **开发过程详细记录**: [DEVLOG.md](./DEVLOG.md)

---

## 核心思路

```
准备内容底稿 → 确定视觉风格 → 生成整体骨架 → 逐页细化内容 → 查找/提供素材 → 反复迭代
```

这不是一次性生成，而是一个 **人机协作的迭代过程**。你负责"要什么"和"对不对"，Claude Code 负责"怎么做"。

---

## 方法论详解

### Step 1：准备内容底稿

在开始之前，先用 Markdown 写好内容大纲。不需要很完美，但要有：
- 整体分几个 Part，每个 Part 讲什么
- 每页 slide 的核心观点（一句话就行）
- 你想讲的案例、素材线索

> 本项目的底稿是一份 Obsidian Markdown 笔记（`openclaw-slides-202603.md`），写好大纲后直接丢给 Claude Code 作为内容输入。

### Step 2：确定视觉风格

**不要让 AI 自己决定风格**，否则会得到千篇一律的"AI 味"设计。你需要主动提供视觉参考：

本项目实际的做法：
1. **提供一份风格规格**：用 YAML 写了一份"日式极简"设计规格（配色、字体、布局模式），作为第一版风格
2. **丢一个 Keynote 做参考**：把之前做过的一份 Keynote 文件路径给 Claude Code，让它提取配色和视觉语言
3. **截图反馈修正**：打开浏览器看效果 → 截图 → 告诉 Claude "背景太灰了，深一点" "加一个科技蓝做装饰色" → 2-3 轮就能定下来

风格确定后，让 Claude Code **把设计规范写进 CLAUDE.md**。这样后续每次生成新页面都会自动遵守同一套视觉语言。

### Step 3：生成整体骨架

把底稿和风格规范准备好后，**一次性生成所有页面的骨架**：

```
把这份 Markdown 内容做成 HTML slides，风格参照 CLAUDE.md 中的设计规范
```

第一版不需要完美——先有骨架，再逐页打磨。本项目第一版 22 页 slides 在一条指令内生成。

### Step 4：逐页细化和迭代

这是花时间最多的阶段。核心模式是 **打开浏览器预览 → 发现问题 → 告诉 Claude Code → 立即修改 → 立即 commit**：

**给概念，让 Claude 实现：**
> "2.1 这页重点是主动交互。传统模式是用户发起，Agent 模式是系统主动。做一个对比的视觉效果"

**给素材来源，让 Claude 去找：**
> "去看一下 https://docs.openclaw.ai/concepts/agent-loop 这个文档，按照里面的架构重画 slide 9 的图"
>
> "读一下 /Users/xxx/专利.docx 这个文件，把里面的方案做成一页 slide"
>
> "去 docko.app 和 looirobot.com 看看，把它们作为主动交互的桌面宠物案例加进去"

**给方向性修正：**
> "2.2 的角度不对，不应该从长期 HRI 的挑战出发，应该从 OpenClaw/Agent 的视角讲具身性——大脑有了，身体是什么？"

**给布局修正（往往需要截图）：**
> "底部内容被截断了"
> "图太小了，不要改宽高比"
> "这两页顺序反了，swap 一下"

### Step 5：素材的处理策略

一个关键经验：**真实素材远比 AI 画的插图有说服力**。

| 素材类型 | 怎么给 Claude Code |
|---------|------------------|
| 本地文件（论文、专利、文档） | 直接给文件路径，让它读取并提取内容 |
| 网页/文档 URL | 给链接，让它 fetch 后理解并转化为 slide 内容 |
| 真实图片（论文 figure、产品截图） | 放到 `assets/` 目录，告诉它用哪张 |
| 架构图/流程图 | 描述概念，让 Claude 用 inline SVG 绘制 |
| 本地项目目录 | 指向真实的代码/配置目录，让它分析并可视化 |

本项目中，Claude Code 实际读取了专利 docx、OpenClaw 的 .openclaw 工作空间目录（含 SOUL.md、对话记录）、论文图片，以及多个在线文档和网站。

## 两个关键配置

### CLAUDE.md：持续演进的设计系统

CLAUDE.md 不是一次写好就不动的。它应该 **随项目推进不断更新**：

- 初始版本：项目概述 + 基本设计规范
- 风格确定后：写入具体配色值、字体、CSS 变量
- 发现问题后：补充约束条件（比如"每页可用高度只有 500-550px，内容多了要拆页"）
- 新增布局模式后：记录可复用的 CSS class

本项目的 CLAUDE.md 在开发过程中被更新了多次，每次都是因为发现了 Claude Code 反复犯的错误。

### frontend-slides Skill：可复用的能力包

Skill 定义了 slides 制作的通用最佳实践（零依赖、Viewport Fitting、审美指导），独立于具体项目。本项目结束后，同一个 skill 被直接复用来做了另一套 Formative Study 主题的 slides。

## 常见问题和应对

| 问题 | 原因 | 解决 |
|------|------|------|
| 底部内容被截断 | 每页 100vh + overflow hidden | 在 CLAUDE.md 中写明高度约束；内容多就拆页 |
| SVG 图表怎么都不满意 | 复杂概念的可视化本来就难 | 接受需要 3-5 轮迭代；描述清楚概念比描述布局更有效 |
| Slide 叙事顺序不对 | 技术逻辑 ≠ 听众理解逻辑 | 从听众视角反复推敲；swap 操作很便宜，不要怕调 |
| 新页面风格不一致 | 页数多了以后 Claude 会"忘记"规范 | 靠 CLAUDE.md 兜底；用统一的 CSS class |
| file:// 打开 CORS 报错 | 浏览器安全策略 | 把外部依赖（p5.js）内联到 HTML 中 |
| 截图/图片让 API 报错 | 大图片或 base64 数据污染上下文 | 图片放 assets/ 目录用路径引用，不要把大图传进对话 |

